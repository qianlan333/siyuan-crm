# Lead Pool V3 Cleanup Notes

## 新表语义

`user_ops_lead_pool_current` 是统一用户运营看板 V2 当前唯一对外读取的引流品成员表。谁在这张表里，谁就是引流品用户。

核心字段语义：

- `mobile`：手机号主线。只要手机号已知，就以手机号作为 canonical key 做合并。
- `external_userid`：企微外部联系人标识。来自绑定关系或自动标签链路，不允许由导入直接写入。
- `is_wecom_added`：是否已经加微。
- `is_mobile_bound`：是否已经完成手机号绑定。
- `huangxiaocan_activation_state`：只表达 `unknown / activated / not_activated`。
- `class_term_no` / `class_term_label`：班期投影字段。
- `first_entry_source` / `last_entry_source`：首入表来源与最近一次入表来源。

`user_ops_lead_pool_history` 是 lead pool 的增量历史表，记录 insert / update / merge / activation patch / deferred verify 等动作。

`user_ops_huangxiaocan_activation_source` 是黄小璨激活状态源表。它只补字段，不负责创建 lead member。

## 3 条入表路径

当前新 lead pool 只认 3 条入表路径：

1. 看板“导入学员”导入
2. 侧边栏手动设置班期
3. 自动打班期标签后，延迟约 10 秒验证成功

`导入黄小璨激活状态` 不创建 lead member，只会：

1. 写 `user_ops_huangxiaocan_activation_source`
2. 按手机号 patch 已存在的 `user_ops_lead_pool_current`

## 手机号主线 merge 规则

- 一旦手机号已知，手机号就是 canonical key。
- external-only 与 mobile-only 记录在 bind-mobile 成功后合并为一条。
- merge 时优先保留：
  - `mobile`
  - `external_userid`
  - `customer_name`
  - `owner_userid`
  - `class_term_no` / `class_term_label`
  - `huangxiaocan_activation_state`
- 合并动作会写 `mobile_bind_merge` history。

## activation 字段语义

新模型只保留三态：

- `unknown`
- `activated`
- `not_activated`

旧 `high_intent` 语义不再属于新版 V2 用户面，不再在 overview / list / export / history 中暴露。

## 为什么旧表先不 drop

当前仍保留旧表，是为了：

- 给回滚留抓手
- 给迁移 helper 留输入来源
- 避免一次性大迁移放大风险

但旧表已经不再是新版 V2 的正常读写主链路。当前策略是：

- 停读
- 停写
- 不再对外暴露旧用户面入口

## 已切到新表的路由

下面这些路由现在都直接读新 lead pool：

- `/api/admin/user-ops/overview`
- `/api/admin/user-ops/list`
- `/api/admin/user-ops/history`
- `/api/admin/user-ops/export`
- `/admin/user-ops/ui`

## 已降级为内部维护用途的旧入口

下面这些旧入口已不再作为 admin V2 正常操作入口：

- `/api/admin/user-ops/reload`
- `/api/admin/user-ops/import-experience-leads`
- `/api/admin/user-ops/backfill-class-term`

它们当前返回 `deprecated_internal_only`，避免继续驱动 legacy `user_ops_pool_*` 路径。
