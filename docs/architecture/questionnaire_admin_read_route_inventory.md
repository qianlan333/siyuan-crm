Read model notes:

- The production read model uses existing PostgreSQL tables: `questionnaires`, `questionnaire_questions`, `questionnaire_options`, `questionnaire_score_rules`, `questionnaire_submissions`, and `questionnaire_submission_answers`.
- Operations reads channel validity and its active QR asset through a composed `ChannelCompletionReadPort`; the implementation remains channel-owned, and the questionnaire context never imports or queries the channel repository directly.
- Operations reads capability state through the platform-owned `PushCapabilityStatusReadService`. An unavailable production data source returns a controlled `503` instead of fixture data.
- No new table migration is required for group 7; `scripts/backfill_questionnaire_admin_read_model.py` provides a dry-run readiness check with `write_executed=false` and does not write data.
- Fixture/local tests may use `local_contract_probe`, but production must never masquerade fixture data as production. Missing production data returns `source_status=production_unavailable`, `read_model_status=unavailable`, `degraded=true`, and `fallback_used=false`.
