# User Ops V2 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a unified CRM user operations board v2 centered on `mobile`, covering unbound trial leads, activation state, class term assignment, and signed status.

**Architecture:** Keep the existing CRM monolith shape. Add a new projection table family for user operations, expose read-first admin APIs under `/api/admin/user-ops/*`, and ship a dedicated admin UI at `/admin/user-ops/ui` using the same backend-rendered admin style already used in `routes.py`.

**Tech Stack:** Flask, existing `routes.py` admin UI pattern, SQLite/PostgreSQL dual schema files, existing CRM binding/contact/tag tables.

## Current Implemented Rules (2026-03-28)

### Final Rule Checklist

1. 手机号是统一用户运营看板 V2 的唯一导入核心键。
2. 班期导入通过 `/api/admin/user-ops/import-mobile-class-terms` 独立处理，同手机号最后一条覆盖班期。
3. 激活导入通过 `/api/admin/user-ops/import-activation-status` 独立处理，同手机号最后一条覆盖激活状态。
4. `external_userid` 不允许通过任何导入直接写入。
5. `是否已加微` 由绑定关系推导：`people.mobile -> external_contact_bindings.person_id -> external_userid`。
6. 看板当前展示的是投影结果，不是所有字段都有独立源模型。

### Source Fields vs Projection Fields

**Source fields**
- `user_ops_experience_leads.mobile`
  - 当前承担体验课导入和班期导入的手机号锚点。
- `user_ops_experience_leads.source_type`
  - 当前区分 `experience_import` 与 `class_term_import`，后续仍需统一语义。
- `user_ops_activation_status_source.mobile`
- `user_ops_activation_status_source.activation_status`
- `user_ops_activation_status_source.activation_remark`

**Projection fields**
- `user_ops_pool_current.mobile`
- `user_ops_pool_current.external_userid`
- `user_ops_pool_current.is_wecom_bound`
- `user_ops_pool_current.class_term_no`
- `user_ops_pool_current.class_term_label`
- `user_ops_pool_current.activation_status`
- `user_ops_pool_current.activation_remark`
- `user_ops_pool_current.current_status`
- `user_ops_pool_current.owner_userid`
- `user_ops_pool_current.customer_name`
- `user_ops_pool_current.source_type`

### Field Semantics

- `class_term_no` / `class_term_label`
  - 当前没有独立班期源表。
  - 班期导入先借助 `user_ops_experience_leads.mobile` 进入手机号池，再在 `reload_user_ops_pool()` 之后回写到 `user_ops_pool_current.class_term_no / class_term_label`。
- `activation_status`
  - 源数据写入 `user_ops_activation_status_source.activation_status`。
  - `reload_user_ops_pool()` 会把它投影到 `user_ops_pool_current.activation_status`。
- `external_userid` / `is_wecom_bound`
  - 只来自既有 CRM 绑定关系，不由导入写入。

### Import Conflict Rules

- 班期导入：
  - 同手机号重复导入时，最后一条覆盖 `class_term_no / class_term_label` 投影值。
- 激活导入：
  - 同手机号重复导入时，最后一条覆盖 `user_ops_activation_status_source.activation_status`，并同步覆盖池子投影。
- 混合导入：
  - 班期导入不会覆盖激活状态。
  - 激活导入不会覆盖班期。
  - 两类导入都不会覆盖绑定推导出的 `external_userid`。

### Weakened But Still Retained Legacy Logic

- `/api/admin/user-ops/list` 仍支持 `current_status`、`owner_userid` 等旧筛选参数，但 W08 后 UI 不再暴露这些控件。
- `/api/admin/user-ops/backfill-class-term` 与 `/api/admin/user-ops/run-deferred-jobs` 仍保留后端接口，但 UI 已不可达。
- 历史 `high_intent` 数据仍可能存在于库里，只是新导入不再允许写入。
- `activation_remark` 仍保留在表结构和投影中，但手机号中心模型下已被弱化，当前新导入统一写空字符串。
- 模板里仍有少量 backfill 相关 CSS 残余样式，但不影响当前 UI 行为。
